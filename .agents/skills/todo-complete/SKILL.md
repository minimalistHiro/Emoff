# TODO Complete

## 概要

ユーザーがTODO項目を指して「完了にして」「廃止にして」と依頼したときに、関連するすべてのドキュメントを一括更新する。

## 手順

### 1. 対象項目の特定

- ユーザーが指定したTODO項目のID（例: A-28）を特定する。
- `/Users/kanekohiroki/Desktop/emoff/TODO.md` を Read ツールで読み込み、該当項目の行を確認する。
- 項目が見つからない場合はユーザーに確認する。
- ユーザーが「完了」「廃止」のどちらを指定しているかを確認する。廃止の場合は理由をユーザーから取得する。

### 2. TODO.md から該当項目を削除

- TODO.md から該当項目の行を Edit ツールで削除する。
- テーブルのヘッダー行や区切り行は削除しない。項目行のみを削除する。
- 「## タスク一覧」内の全項目が削除されて空テーブルになる場合は、テーブル自体を残す（ヘッダー行は維持）。

### 3. COMPLETED.md に追記

- `/Users/kanekohiroki/Desktop/emoff/COMPLETED.md` を Read ツールで読み込む。
- 今日の日付（YYYY-MM-DD）のセクション（`## YYYY-MM-DD`）が存在するか確認する。
  - 存在する場合：そのセクションのテーブル末尾に項目を追記する。
  - 存在しない場合：既存の最新日付セクションの直前に新しい日付セクションを作成し、項目を追記する（降順: 新しい日付が上）。
- テーブルのカラムは `| ID | 優先度 | 内容 | 状態 |` とする（既存COMPLETED.mdの形式に合わせる）。
- **完了の場合**: 状態欄に `[x] 完了（YYYY-MM-DD）` を記載する。実装内容の補足がある場合は「内容」列の末尾に `— 補足説明` の形で追記する。
- **廃止の場合**: 状態欄に `[x] 廃止（YYYY-MM-DD）` を記載し、内容列の末尾に `— **廃止理由**: 理由` を追記する。

### 4. QAチェックリストの更新（完了項目のみ、qa/ ディレクトリが存在する場合）

- **廃止項目の場合はこのステップをスキップする。**
- `/Users/kanekohiroki/Desktop/emoff/qa/` ディレクトリが存在しない場合もスキップする（emoff は QA 運用未整備のうちは追記対象なし）。
- ディレクトリが存在する場合のみ以下を実行する：

  1. `pubspec.yaml` の `version:` フィールドから現在のアプリバージョン（例: `1.0.0+1`）を取得する。
  2. `/Users/kanekohiroki/Desktop/emoff/qa/QA_V{バージョン}.md` を読み込む。ファイルが存在しない場合は以下のフォーマットで新規作成する：

     ```
     # QA チェックリスト — v{バージョン} (Build {ビルド番号})

     バージョン {バージョン} の主要変更点に対する探索テスト項目。
     実際にアプリを操作しながら確認する。気になった点・不具合はTODOや要検討事項に直接追記する。

     - **対象バージョン**: {バージョン}+{ビルド番号}
     - **テスト項目作成開始日**: {今日の日付}
     - **テストID形式**: `V{バージョン数字のみ}-NNN`
     - **方針**: 完了したTODO項目に基づく主要変更点の動作確認

     ---

     ## チェック項目

     - [ ] `V{バージョン数字のみ}-001` テスト内容（TODO ID）
     ```

  3. **チェック項目の形式は `- [ ] `V{バージョン数字のみ}-NNN` テスト内容（TODO ID）` とする。**
  4. 既存のチェック項目の最大番号の次から採番する。
  5. テスト項目の内容は、TODO項目の内容をベースに、ユーザーが操作して確認できる内容に要約する。末尾に `（TODO ID）` を付与する。
  6. `/Users/kanekohiroki/Desktop/emoff/QA_CHECKLIST.md` のバージョン別チェックリストテーブルに該当バージョン行があれば項目数を +1 する。無ければ追記候補としてユーザーに通知する（このスキルでは新規行を勝手に追加しない）。
  7. テスト結果（Pass/Fail/Skip）が確定したら `QA_RESULTS.md` 側に記録する運用なので、本スキルで `QA_RESULTS.md` は更新しない。

### 5. 計画書のステータス更新（計画書がある場合のみ）

- TODO項目の内容に `計画書:` の記載がある、または `/Users/kanekohiroki/Desktop/emoff/plans/in_progress/` に関連する計画書ファイルが存在する場合のみ以下を実行する。
- 計画書の `- **ステータス**:` 行を `- **ステータス**: 実装完了（YYYY-MM-DD）` に更新する。
- 計画書内のタスクチェックリスト（`- [ ]`）がある場合は、実装済みの項目を `- [x]` に更新する。
- 計画書ファイルを `plans/in_progress/` から `plans/completed/` へ移動する（`mv` コマンド使用）。
- `plans/completed/` ディレクトリが存在しない場合は作成する。

### 6. スプレッドシート同期

ユーザーがスプレッドシート運用を有効にしている場合、以下を実行する：

```bash
python3 scripts/emoff_sheets.py todo
python3 scripts/emoff_sheets.py qa
```

`qa/` が存在しない、または該当バージョンのQAファイルがない場合はQA同期をスキップし、その理由を報告する。

### 7. 結果報告

更新したファイルの一覧（TODO.md / COMPLETED.md / QAチェックリスト / 計画書）を日本語で報告する。

## 対象となる依頼例

- こちらを完了にして
- このタスクを完了にしてください
- A-28を完了にして
- 〇〇を完了としてください
- このTODOを廃止にして
- 〇〇は不要になったので廃止にして
- この項目は完了しました

## 注意事項

- TODO.md からは項目行を **完全に削除** する。完了マークを付けて残すのではなく、削除してCOMPLETED.mdへ移動する。
- COMPLETED.md の日付セクションは降順（新しい日付が上）を維持する。
- 廃止項目はQAチェックリストに追加しない。
- 1つの完了TODO = 1つのQAチェック項目を基本とする。
- 計画書の移動先ディレクトリ `plans/completed/` が存在しない場合は作成する。
- スプレッドシート連携を行う場合は `scripts/credentials.json` と `EMOFF_SPREADSHEET_ID` または `scripts/sheets_config.json` が必要。
