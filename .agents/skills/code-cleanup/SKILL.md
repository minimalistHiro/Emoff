# コード整理（code-cleanup）

## 概要

`git diff main..HEAD` で今回の作業ブランチで変更された `.dart` ファイルのみを対象に、パフォーマンス問題・コード品質問題を検出し、ユーザー承認後に修正する。

## 手順

### フェーズ0：仕様の把握（最初に必ず実行）

コード修正・リファクタリングで既存の仕様を意図せず変更してしまうことを防ぐため、**分析・修正を始める前に** 以下のファイルを読み取り、アプリの仕様を把握する。

1. `/Users/kanekohiroki/Desktop/emoff/APP_SCREENS.md` — 画面仕様・遷移フロー
2. `/Users/kanekohiroki/Desktop/emoff/SCREEN_DESIGN.md` — 画面詳細設計
3. `/Users/kanekohiroki/Desktop/emoff/FIRESTORE.md` — Firestoreスキーマ構造
4. `/Users/kanekohiroki/Desktop/emoff/UI_UX.md` — UI/UXコンポーネント規約・デザインガイドライン
5. `/Users/kanekohiroki/Desktop/emoff/BUSINESS_MODEL.md` — 課金プラン仕様・Free/Pro の境界（プラン判定ロジックに触れる場合のみ）

これらの仕様を把握した上で、以降のフェーズに進む。**リファクタリングはあくまで内部処理の書き換えであり、画面仕様・データ構造・UI/UX規約・プラン判定ロジックを変えてはならない。**

### フェーズ1：未コミット変更のコミット

フェーズ2の `git diff main..HEAD` を正常に機能させるため、未コミットの変更がある場合は先にコミットする。

1. `git status --short` で未コミットの変更（ステージ済み・未ステージ・未追跡ファイル）があるか確認する。
2. **変更がない場合** → このフェーズをスキップしてフェーズ2に進む。
3. **変更がある場合** → 以下を実行する：
   1. 変更内容に基づいてコミットメッセージを自動生成する（変更されたファイルの種類・機能領域から簡潔に要約）。
   2. ステージング対象を `git status` で確認した上で、機密ファイル（`.env` / `Google*-Info.plist` 等）を含まないように具体ファイル名で `git add <files>` する。
   3. `git commit` でローカルにコミットする（**pushはしない**）。
   4. コミットした旨をユーザーに報告する。

> **注意**: このコミットはローカルのみ。pushは行わない。ユーザーが後から `github-commit-push-rules` を実行した際に、整理分のコミットと合わせてpushされる。

### フェーズ2：差分の特定

1. `git diff --name-only main..HEAD` で変更ファイル一覧を取得する。
2. **差分がない場合** → 「mainとの差分がないため、整理対象がありません」と報告して **終了**。
3. 差分がある場合 → `.dart` ファイルのみを対象として抽出する（マークダウン・設定ファイル・iOS/Android ネイティブ設定は対象外）。

### フェーズ3：変更ファイルの分析

対象の `.dart` ファイルを読み込み、以下のアンチパターンを検出する。

#### パフォーマンス問題

- **setState() の過剰呼び出し**: タイマー・アニメーションリスナー・ストリーム listen 内での直接呼び出しによる過剰な再描画。
- **Firestore リスナーの過多／解放漏れ**: `snapshots()` を不必要に多く購読、`StreamSubscription.cancel()` 漏れ。
- **dispose() でのリソース解放漏れ**: Timer、StreamSubscription、AnimationController、TextEditingController、FocusNode 等の未解放。
- **Image.network のキャッシュ管理不在**: `cached_network_image` 未使用、`cacheWidth`/`cacheHeight` 未指定（特にアイコン・サムネイル系）。
- **無制限に増殖するキャッシュ**: サイズ上限なしの Map/List がメモリを圧迫する箇所。
- **重い処理の繰り返し呼び出し**: `build()` 内での重い計算・コレクション再生成、`ListView.builder` を使わずに全要素を一度に描画している箇所。
- **AI 変換 API の冗長呼び出し**: 同一テキストに対する変換要求の重複・上限制御抜け。

#### コード品質

- **1000行超の巨大ファイル**: 分割を推奨。
- **同一処理の重複（DRY違反）**: 共通化すべきロジックの検出（特に複数画面で重複するフォーマッタ・バリデーション・Firestore 参照）。
- **不要な rebuild**: `Provider` / `Riverpod` の `watch` 範囲が広すぎる、`Consumer` 粒度の粗さ。
- **未使用の import・変数・メソッド**: 不要コードの検出。
- **共通ウィジェット未使用**: UI_UX.md に列挙されているカスタムウィジェットを使わず素の Widget を使っている箇所（`CustomLoadingIndicator` 等）。

### フェーズ4：報告（ユーザー確認）

検出結果を以下の形式でテーブル表示する：

| 重大度 | 対象ファイル:行番号 | 問題の内容 | 推奨される修正方針 |

重大度は **最高 / 高 / 中 / 低** の4段階。

報告後、**「修正してよいですか？」** とユーザーに確認する。勝手に修正しない。

### フェーズ5：修正（承認後のみ実行）

1. ユーザーが承認した項目のみコードを修正する。
2. 修正内容をファイルごとに報告する。
3. 修正で対応しきれない大きな課題がある場合は `TODO.md` に追加する。
   - `todo-add` スキルの採番ルールに従い、TODO.md と COMPLETED.md の最大IDの次から採番する。

### フェーズ6：ビルド／解析確認

1. `flutter analyze` を実行し、新規 issue が増えていないことを確認する。
2. `flutter build apk --debug` または `flutter build ios --debug --no-codesign` を実行し、ビルドエラーがないことを確認する（プラットフォーム選択はユーザーに確認）。
3. **成功** → 「ビルド確認OK」と報告して終了。
4. **エラー発生** → エラー内容を修正し、再度実行する。エラーが解消するまで繰り返す。

## 対象となる依頼例

- コードを整理して
- コードをクリーンアップして
- コードを最適化して
- パフォーマンスを改善して
- 無駄な処理を見直して
- リファクタリングして

## 注意事項

- **フェーズ1のコミット後も差分がない場合は何も行わない**。
- **対象は `.dart` ファイルのみ**（マークダウン・設定ファイル・ネイティブ設定は対象外）。
- **フェーズ4で必ずユーザー確認を挟む**。検出結果を報告せずに修正を始めてはならない。
- `UI_UX.md` はフェーズ0で常に読み取る（CLAUDE.md のルールを包含）。
- 修正はアプリの動作を変えない範囲（リファクタリング）に留める。機能追加・仕様変更は行わない。
- フェーズ0で把握した仕様（画面仕様・Firestoreスキーマ・UI/UX規約・プラン判定ロジック）に影響する変更は行わない。
- `git add -A` は **使わない**。`.env` や `Google*-Info.plist` 等の機密ファイルを巻き込まないよう、具体ファイル名でステージングすること。
