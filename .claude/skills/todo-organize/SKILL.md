# TODO 設計組み合わせ分析

## 概要

TODO.md の各項目を分析し、同時に設計・実装した方が効率的な関連項目の組み合わせを特定する。特定した組み合わせは各テーブルの「備考」列に記載する。

## 手順

### 1. TODO.md を読み込む

`/Users/kanekohiroki/Desktop/emoff/TODO.md` を Read ツールで読み込み、全項目を把握する。

### 2. 関連項目の組み合わせを分析する

以下の観点で、同時に設計すべき項目を特定する：

- **データモデル共通**: 同じ Firestore コレクション・ドキュメント・フィールドを変更する項目（例: `users/{uid}` / `chats/{chatId}/messages` 等）
- **Cloud Functions 共通**: 同じトリガー（onCreate / onUpdate / onSchedule / Webhook 等）を共有する項目
- **UI コンポーネント共通**: 同じ画面・共通ウィジェット・ボトムシート・ダイアログに影響する項目
- **チャット／AI変換ロジック共通**: メッセージ送信・AI変換プレビュー・トーン選択・上限制御等の同一フローに関わる項目
- **課金／プラン共通**: RevenueCat / プラン判定 / Free 上限制御 等のサブスク関連で重なる項目
- **軽微な修正のまとめ**: 個別では小さいが、同種の修正をまとめて一括対応できる項目

### 3. 分析結果をユーザーに提示する

組み合わせの一覧を以下の形式でユーザーに提示し、承認を得る：

```
| グループ | 対象項目 | 理由 |
|---------|---------|------|
| グループ名 | A-XX, A-YY | 共通する設計要素の説明 |
```

### 4. 備考欄に記載する

ユーザーの承認後、TODO.md の「タスク一覧」テーブルに対して以下を行う：

1. **備考列がない場合**: テーブルヘッダーに `| 備考 |` 列を追加し、区切り行にも `|---|` を追加する（既存行の末尾には空の備考セル `| |` を追加する）。
2. **備考列がある場合**: 既存の備考列を活用する。
3. 各項目の備考セルに `設計組み合わせ: A-XX, A-YY（理由）` の形式で記載する。
4. 組み合わせに該当しない項目の備考セルは空のままにする。
5. すでに状態が `[ ] 計画済み` や `[ ] 進行中` の項目は、組み合わせ対象から除外する。

### 5. 結果を報告する

更新した組み合わせの一覧を日本語で報告する。

## 対象となる依頼例

- TODOを整理して
- TODOの組み合わせを分析して
- 関連タスクをまとめて
- 設計組み合わせを更新して
- 一緒に設計すべき項目を教えて
- TODOをグルーピングして

## 注意事項

- 組み合わせの記載前に、必ずユーザーに分析結果を提示して承認を得ること。
- すでに `[ ] 計画済み` や `[ ] 進行中` の項目は組み合わせ対象から除外する。
- 備考欄の既存内容がある場合は上書きせず、ユーザーに確認する。
- 備考欄のフォーマット: `設計組み合わせ: A-XX, A-YY（理由）`。
- 組み合わせは **双方向で記載する**（A-XX→A-YY なら、A-YY→A-XX も記載する）。
